# Jam 提出用スクリーンショット撮影計画

README と Jam 提出ページで、作品の「回答ではなく根拠構造を探索する」体験が伝わるように、以下の画面を撮影します。実画像はまだリポジトリに配置していないため、**提出前にこの計画に沿った画像へ差し替えます**。

## 必須カット

| 優先度 | 画面 | 撮影状態 | 伝える内容 | 推奨ファイル名 |
| --- | --- | --- | --- | --- |
| 1 | Case000 調査画面 | 複数ノードを確認し、中央の Memory Network、ノード詳細、判断条件が同時に見える状態 | 記憶ネットワークを探索する中核体験 | `docs/images/case000-investigation.png` |
| 2 | 結果画面 | 判断 A / B / C のいずれかを提出し、行政処理ログと次回予告が見える状態 | 調査結果が裁定と価値判断へ接続すること | `docs/images/case000-result.png` |
| 3 | タイトル画面 | 初回起動直後 | 作品名、監査端末、サイバー世界観 | `docs/images/title.png` |

## 追加推奨カット

| 画面 | 撮影状態 | 伝える内容 | 推奨ファイル名 |
| --- | --- | --- | --- |
| 事件選択 | Case000 と Case001 preview が同時に見える状態 | 実装範囲と Case001 が予告のみであること | `docs/images/case-select.png` |
| 調査画面・判断開放 | 必要ノード確認、1件以上のピン留め、1件以上の矛盾タグ付けを完了した状態 | 根拠を揃えて判断を開放するルール | `docs/images/judgment-unlocked.png` |
| 最終判断 | A / B / C の3選択肢が見える状態 | 正解当てではなく行政判断を選ぶ構造 | `docs/images/decision.png` |

## 撮影ルール

- 本番相当の Vercel 公開 URL、または `npm run dev` の表示を使用します。
- 16:9、幅 1440px 以上を推奨します。
- ブラウザの開発者ツール、個人情報、不要なタブや通知を写しません。
- 文字が読める倍率を優先し、サイバー演出だけで情報構造が潰れないようにします。
- 調査画面では7ノードが確認できる構図を優先します。
- 結果画面では裁定、処理内容、価値、監査注記、Case001 予告のうち複数セクションが見えるようにします。

## README への反映

画像を `docs/images/` に配置した後、README のスクリーンショット欄に最低でも次の2点を掲載します。

1. `case000-investigation.png`
2. `case000-result.png`

画像が未配置の現在は、README に撮影計画へのリンクと「提出前に実画像へ差し替える」旨を表示します。画像追加後はその注意書きを削除し、画像と短いキャプションに置き換えます。
